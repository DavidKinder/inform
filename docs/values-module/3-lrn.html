To parse real numbers written as decimal expansions, or in scientific or engineering notation.

§1. Rvalues for the "real number" kind follow the IEEE-754 single-precision encoding for floating-point numbers.

\(\pi\) and \(e\) get their own special treatment here, if written in lower case. But \(\gamma\), always in the bronze medal position among mathematical constants, misses out.

§3. The following function, adapted from code originally written for the Inform 6 compiler by Andrew Plotkin, returns the IEEE-754 single-precision encoding of a floating-point number.

If the magnitude is too large (beyond about 3.4e+38), this returns plus or minus infinity; if the magnitude is too small (below about 1e-45), this returns a zero value. If any of the inputs are NaN, this returns NaN. 

int LiteralReals::construct_float(int signbit, double intv, double fracv, int expo) {
    double absval = (intv + fracv) * LiteralReals::ten_to_the(expo);
    int sign = (signbit ? ((int) 0x80000000) : 0x0);

    latest_constructed_real = absval; if (signbit) latest_constructed_real = -absval;

    if (isinf(absval)) Return plus or minus infinity3.5;
    if (isnan(absval)) Return plus or minus Not-a-Number3.6;

    double mant = frexp(absval, &expo);
    Normalize mantissa to be in the range [1.0, 2.0)3.1;

    if (expo >= 128) Return plus or minus infinity3.5;
    if (expo < -126) Denormalize this very small number3.2
    else if (!(expo == 0 && mant == 0.0)) Denormalize this non-zero number3.3;

    int fbits = 0;
    Set fbits to the mantissa times 2 to the 23, rounded to the nearest integer3.4;

    return (sign) | ((int)(expo << 23)) | (fbits);
}

§3.4. At this point the mantissa is a number in the range [0, 1), so multiplying it by 2 to the 23 will produce the bottom 22 bits of our answer. The case we have to be wary of is where the carry propagates out of a string of 23 1 bits, that is, where we end up with bit 23 set as well: in that case, the exponent increments, and we round to that power of 10 exactly.

Note that 2 to the 23 is 8388608.

Set fbits to the mantissa times 2 to the 23, rounded to the nearest integer3.4 = 

    fbits = (int) ((mant*8388608.0) + 0.5);  round to nearest integer
    if (fbits >> 23) {
        fbits = 0;
        expo++; if (expo >= 255) Return plus or minus infinity3.5;
    }
  • This code is used in §3.

§4. The following returns 10 to the given power. 

define POW10_RANGE 8

double LiteralReals::ten_to_the(int expo) {
    static double powers[POW10_RANGE*2+1] = {
        0.00000001, 0.0000001, 0.000001, 0.00001, 0.0001, 0.001, 0.01, 0.1,
        1.0,
        10.0, 100.0, 1000.0, 10000.0, 100000.0, 1000000.0, 10000000.0, 100000000.0
    };

    double res = 1.0;

    if (expo < 0)
        for (; expo < -POW10_RANGE; expo += POW10_RANGE)
            res *= powers[0];
    else
        for (; expo > POW10_RANGE; expo -= POW10_RANGE)
            res *= powers[POW10_RANGE*2];

    return res * powers[POW10_RANGE+expo];
}

§5. The problem message for engineering notation should only appear once: 

int e_notation_problem_issued = FALSE;

void LiteralReals::wrong_notation(wording W) {
    if (e_notation_problem_issued == FALSE) {
        e_notation_problem_issued = TRUE;
        Problems::quote_source(1, current_sentence);
        Problems::quote_wording(2, W);
        StandardProblems::handmade_problem(Task::syntax_tree(), _p_(PM_WantonEngineering));
        Problems::issue_problem_segment(
            "In %1, you write '%2', which looks to me like the engineering "
            "notation for a real number - I'm guessing that the 'e' means "
            "exponent, so for example, 1E+6 means 1000000. Inform writes "
            "numbers like this as 1 x 10^6, or if you prefer 1.0 x 10^6; "
            "or you can use a multiplication sign instead of the 'x'.");
        Problems::issue_problem_end();
    }
}

§6. This upgrades a literal K_number to a literal K_real_number if necessary, as it can be when performing arithmetic. Note that we do this not by running an integer-to-real conversion inside Inform, but just by re-parsing it in a real context: the result is therefore identical to what would have been parsed had the literal in question been expected to be real all along. 

int promoting_reals = FALSE;
int LiteralReals::promote(void) {
    return promoting_reals;
}
parse_node *LiteralReals::promote_number_if_necessary(parse_node *value, kind *to) {
    int s = promoting_reals;
    promoting_reals = TRUE;
    kind *from = Specifications::to_kind(value);
    if ((Kinds::eq(from, K_number)) && (Kinds::eq(to, K_real_number))) {
        wording W = Node::get_text(value);
        if (<s-literal-real-number>(W)) value = <<rp>>;
    }
    promoting_reals = s;
    return value;
}
